# Component Props

help components remain reusable.

## Declaring Props

### Basic Example

```tsx
type AvatarProps = {
   name: string;
   imageUrl?: string;
};

function Avatar({ name, imageUrl = "/placeholder.png" }: AvatarProps) {
   return (
      <img
         alt={name}
         src={imageUrl}
         className='h-10 w-10 rounded-full border border-border'
      />
   );
}
```

### Practical Example

```tsx
type TaskCardProps = {
   task: Task;
   onStatusChange?: (taskId: string, status: Task["status"]) => void;
};

export function TaskCard({ task, onStatusChange }: TaskCardProps) {
   const handleClick = () => {
      if (!onStatusChange) return;
      onStatusChange(task.id, task.status === "done" ? "todo" : "done");
   };

   return (
      <article className='rounded-lg border bg-card p-4'>
         <header className='flex items-center justify-between'>
            <h3 className='text-base font-medium'>{task.title}</h3>
            <button
               type='button'
               className='text-sm text-primary'
               onClick={handleClick}
            >
               Mark {task.status === "done" ? "To-do" : "Done"}
            </button>
         </header>
      </article>
   );
}
```

## ‚úÖ Best Practices

### 1. Document Prop Contracts

**Why:** Future maintainers understand usage quickly.

```md
TaskCard props:

-  `task`: Task object with id/title/status.
-  `onStatusChange`: Optional callback triggered by CTA.
```

### 2. Use Specific Types

**Why:** `Record<string, any>` hides intent; prefer precise shapes.

```ts
type Status = "todo" | "in-progress" | "done";
```

---

## ‚ùå Common Mistakes

### 1. Mutating Props

**Problem:** Props should be read-only.

```tsx
// Bad
props.task.status = "done";
```

**Solution:** Trigger state updates in the parent.

### 2. Passing Entire App State

**Problem:** Components become tightly coupled.

**Solution:** Send only what's needed or derive data locally.
