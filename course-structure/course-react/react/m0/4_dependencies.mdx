## What is npm?

**npm** (Node Package Manager) is the world's largest software registry,
containing over 2 million packages. It's the standard way to manage JavaScript
dependencies.

**Industry Standard**

-  Used by 97% of JavaScript developers
-  Over 10 billion downloads per week

**Dependency Management**

-  Install libraries with one command
-  Automatically manage versions
-  Share your project setup easily

**Build Tools**

-  Run development servers
-  Build production code
-  Execute custom scripts

## Understanding package.json

### What is package.json?

The `package.json` file is the **heart of your Node.js project**. It contains
metadata about your project and lists all dependencies.

### Anatomy of package.json

```json
{
   "name": "taskflow-pro",
   "version": "0.1.0",
   "type": "module",
   "description": "A modern task management application built with React",
   "author": "Your Name",
   "license": "MIT",

   "scripts": {
      "dev": "vite",
      "build": "vite build",
      "preview": "vite preview",
      "lint": "eslint . --ext js,jsx",
      "format": "prettier --write \"src/**/*.{js,jsx,json,css}\""
   },

   "dependencies": {
      "react": "^18.2.0",
      "react-dom": "^18.2.0",
      "react-router-dom": "^6.16.0"
   },

   "devDependencies": {
      "@vitejs/plugin-react": "^4.0.4",
      "vite": "^4.4.9",
      "eslint": "^8.49.0",
      "prettier": "^3.0.3",
      "tailwindcss": "^3.3.3",
      "autoprefixer": "^10.4.15",
      "postcss": "^8.4.29"
   }
}
```

### Key Sections Explained

#### **Name and Version**

```json
"name": "taskflow-pro",
"version": "0.1.0"
```

-  **name**: Unique identifier for your project
-  **version**: Semantic versioning (major.minor.patch)

#### **Scripts**

```json
"scripts": {
  "dev": "vite",
  "build": "vite build"
}
```

-  Custom commands you can run with `npm run`
-  `npm run dev` starts development server
-  `npm run build` creates production build

#### **Dependencies**

```json
"dependencies": {
  "react": "^18.2.0"
}
```

-  Libraries needed for the app to **run** in production
-  Installed automatically when someone clones your project

#### **DevDependencies**

```json
"devDependencies": {
  "vite": "^4.4.9",
  "eslint": "^8.49.0"
}
```

-  Tools needed for **development** only
-  Not included in production build (smaller bundle size)

## Version Numbers Explained

### Semantic Versioning (SemVer)

```
^18.2.0
 ‚Üë  ‚Üë ‚Üë
 ‚îÇ  ‚îÇ ‚îî‚îÄ‚îÄ‚îÄ Patch: Bug fixes (18.2.1)
 ‚îÇ  ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ Minor: New features, backwards compatible (18.3.0)
 ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ Major: Breaking changes (19.0.0)
```

### Version Symbols

```json
"react": "18.2.0"   // Exact version only
"react": "^18.2.0"  // Compatible updates (18.x.x)
"react": "~18.2.0"  // Patch updates only (18.2.x)
"react": "*"        // Any version (dangerous!)
```

**Best Practice:** Use `^` (caret) for most dependencies. It allows minor
updates while preventing breaking changes.

## Installing Dependencies

### Initial Installation

When you clone a project:

```bash
# Install all dependencies
npm install
# or
npm i

# This installs everything in package.json
```

### Adding New Dependencies

#### Production Dependencies

```bash
# Add to dependencies (production)
npm install package-name

# Example: Add React Router
npm install react-router-dom

# Result in package.json:
"dependencies": {
  "react-router-dom": "^6.16.0"
}
```

#### Development Dependencies

```bash
# Add to devDependencies (development only)
npm install --save-dev package-name
# or
npm install -D package-name

# Example: Add ESLint
npm install --save-dev eslint

# Result in package.json:
"devDependencies": {
  "eslint": "^8.49.0"
}
```

### Removing Dependencies

```bash
# Remove a package
npm uninstall package-name

# Example
npm uninstall lodash
```

### Updating Dependencies

```bash
# Check for outdated packages
npm outdated

# Update all packages (respecting version rules)
npm update

# Update specific package
npm update react

# Update to latest version (ignoring version rules)
npm install react@latest
```

## package-lock.json

### What is it?

`package-lock.json` is automatically generated and contains:

-  Exact versions of all dependencies
-  Dependency tree
-  Package integrity checksums

### Why it matters

**Consistency Across Environments**

```
Your computer:    React 18.2.0
Teammate's PC:    React 18.2.0  ‚úÖ Same version
Production:       React 18.2.0  ‚úÖ Same version
```

**Without package-lock.json:**

```
Your computer:    React 18.2.0
Teammate's PC:    React 18.3.1  ‚ùå Different!
Production:       React 18.2.5  ‚ùå Different!
```

**Best Practice:** Always commit `package-lock.json` to version control

## ‚úÖ Best Practices

### 1. Always Use package-lock.json

**Why:** Ensures consistent dependencies across all environments

```bash
# Good: Install using package-lock.json
npm install

# Bad: Delete package-lock.json and reinstall
rm package-lock.json && npm install  # ‚ùå Don't do this!
```

### 2. Separate Production and Development Dependencies

**Why:** Smaller production bundle size, clearer purpose

```json
// Good: Clear separation
{
  "dependencies": {
    "react": "^18.2.0",           // App needs this to run
    "react-router-dom": "^6.16.0" // App needs this to run
  },
  "devDependencies": {
    "vite": "^4.4.9",            // Only for development
    "eslint": "^8.49.0"          // Only for development
  }
}

// Bad: Everything in dependencies
{
  "dependencies": {
    "react": "^18.2.0",
    "vite": "^4.4.9",  // ‚ùå This is a dev tool!
    "eslint": "^8.49.0" // ‚ùå This is a dev tool!
  }
}
```

### 3. Keep Dependencies Updated

**Why:** Security patches, bug fixes, new features

```bash
# Check for updates weekly
npm outdated

# Update patch and minor versions
npm update

# Review changes before major updates
npm install react@latest --save-exact
```

## ‚ùå Common Mistakes

### 1. Installing Packages Globally

**Problem:** Version conflicts and team inconsistency

```bash
# Bad: Install globally
npm install -g create-react-app
npm install -g eslint

# Problems:
# - Different versions on different machines
# - Not tracked in package.json
# - Harder for team collaboration

# Good: Use npx or install locally
npx create-react-app my-app
npm install --save-dev eslint
```

**Solution:** Use `npx` for one-time commands, install tools locally as
devDependencies

### 2. Committing node_modules

**Problem:** Huge repository size, merge conflicts

```bash
# Bad: Commit node_modules
git add .
git commit -m "Added dependencies"
# Result: 200MB+ added to repo

# Good: node_modules in .gitignore
echo "node_modules" >> .gitignore
git add package.json package-lock.json
git commit -m "Added dependencies"
# Result: Only package files committed
```

**Solution:** Always have `node_modules` in `.gitignore`. Let npm install handle
it.

### 3. Ignoring Dependency Warnings

**Problem:** Security vulnerabilities and broken dependencies

```bash
# ‚ö†Ô∏è npm install shows warnings:
found 3 vulnerabilities (1 moderate, 2 high)

# Bad: Ignore and continue
# Security risks, potential bugs

# Good: Investigate and fix
npm audit
npm audit fix
# or
npm update
```

**Solution:** Run `npm audit` regularly and fix vulnerabilities promptly
