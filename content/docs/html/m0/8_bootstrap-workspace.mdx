# Bootstrap the Portfolio Pulse Workspace

## Learning Objectives

By the end of this lesson, you will:

-  [ ] Organize the `portfolio-pulse/` repository with clear folders.
-  [ ] Extract shared snippets into partials for reuse.
-  [ ] Set up documentation and assets directories with naming conventions.
-  [ ] Capture version control rituals and branching strategy.

## Project Context

Even a single-page site benefits from structure. As you layer CSS, JavaScript,
and deployment tooling, a tidy workspace prevents confusion. You will mirror
production standards by separating content, assets, and documentation.

---

## Recommended Structure

### Core Directories

-  `public/` for assets served as-is (images, icons, fonts)
-  `src/` for HTML partials, includes, and templates (if using build tooling)
-  `docs/` for briefs, notes, and process artifacts
-  `notes/` for personal reflections and validation logs

### Naming Conventions

Use kebab-case for files (`hero-section.html`, `about-story.html`) and keep
commit messages consistent (`feat:`, `docs:`, `chore:`, `fix:`).

## âœ… Best Practices

### 1. Centralize Repeatable Snippets

**Why:** Extracting `<head>` and `<footer>` into partials keeps metadata and
contact info consistent across pages.

```html
<!-- src/partials/head.html -->
<head>
   ...
</head>
```

### 2. Adopt a Branching Strategy Early

**Why:** Even solo builders benefit from feature branches for traceability and
safe experimentation.

```bash
git checkout -b feat/hero-structure
# work...
git checkout main
git merge feat/hero-structure
```

### 1. Mixing Source and Build Outputs

**Problem:** Serving files from both `src/` and root confuses deployment setups.

```txt

```

### 2. Skipping README or Project Documentation

**Problem:** Collaborators (or future you) have no idea how to run or extend the
project.

```md
<!-- Bad: empty README -->
```

**Solution:**

```md

-  Setup instructions
-  Deployment commands
-  Content update workflow
```

makes onboarding straightforward.
