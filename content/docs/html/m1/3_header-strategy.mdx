# Crafting the Portfolio Header

## Learning Objectives

By the end of this lesson, you will:

-  [ ] Build a semantic header that communicates the brand quickly.
-  [ ] Combine navigation and call-to-action elements responsibly.
-  [ ] Provide a mobile-friendly menu using native details/summary.
-  [ ] Document responsive considerations for future CSS work.

## Project Context

The header is the first impression. It must clarify who you are, where to go,
and how to get in touch‚Äîall while staying accessible. A well-structured header
also makes future responsiveness work easier, since semantic grouping aligns
with design system components.

---

## Header Composition

### Branding

Include a logo or typographic mark inside a `<div>` or `<span>` that clearly
identifies the site. For screen readers, use `aria-label` or visually hidden
text.

### Navigation

Keep navigation inside `<nav>` with a descriptive `aria-label`. Provide both
inline and mobile-friendly patterns.

### Call-to-Action

Include a primary button linking to `#contact` or a scheduling tool. For
responsive behavior without JavaScript, consider `<details>` + `<summary>`.

### Basic Example

```html
<header role="banner">
   <div class="brand">
      <a href="#top">Portfolio Pulse</a>
   </div>
   <nav aria-label="Primary navigation">...</nav>
   <a class="btn-primary" href="#contact">Book a call</a>
</header>
```

### Practical Example

```html
<header class="site-header" role="banner">
   <div class="brand">
      <a href="#top" aria-label="Portfolio Pulse">
         <span aria-hidden="true">PP</span>
         <span class="sr-only">Portfolio Pulse</span>
      </a>
   </div>
   <nav aria-label="Primary navigation" class="nav-desktop">
      <ul>
         <li><a href="#highlights">Work</a></li>
         <li><a href="#services">Services</a></li>
         <li><a href="#testimonials">Proof</a></li>
         <li><a href="#contact">Contact</a></li>
      </ul>
   </nav>
   <details class="nav-mobile">
      <summary aria-haspopup="listbox">Menu</summary>
      <nav aria-label="Mobile navigation">
         <ul>
            <li><a href="#highlights">Work</a></li>
            <li><a href="#services">Services</a></li>
            <li><a href="#testimonials">Proof</a></li>
            <li><a href="#contact">Contact</a></li>
         </ul>
      </nav>
   </details>
   <a class="btn-primary" href="#contact">Book a discovery call</a>
</header>
```

## ‚úÖ Best Practices

### 1. Reuse Navigation Lists Across Variants

**Why:** Maintaining a single source of truth for nav items prevents drift
between desktop and mobile experiences.

```md
-  Use server-side includes or a build process to reuse nav lists.
-  If duplication is temporary, add a TODO referencing the shared file.
```

### 2. Provide Accessible Menu Toggles

**Why:** `<details>` and `<summary>` give you keyboard and screen reader support
out of the box.

```html
<details>
   <summary>Menu</summary>
   <ul>
      ...
   </ul>
</details>
```

---

## ‚ùå Common Mistakes

### 1. Relying on Icon Fonts Without Text Equivalents

**Problem:** Icons without labels render meaningless to screen readers.

```html
<!-- Bad: icon-only brand link -->
<a href="#top"><span class="icon-logo"></span></a>
```

**Solution:**

```html
<a href="#top" aria-label="Portfolio Pulse">
   <span class="icon-logo" aria-hidden="true"></span>
   <span class="sr-only">Portfolio Pulse</span>
</a>
```

### 2. Forgetting About Focus in Mobile Menus

**Problem:** If tab order does not move inside the opened menu, keyboard users
get stuck.

```html
<!-- Bad: using div with tabindex for menu toggle -->
<div tabindex="0">Menu</div>
```

**Solution:**

```html
<summary aria-haspopup="listbox">Menu</summary>
```
