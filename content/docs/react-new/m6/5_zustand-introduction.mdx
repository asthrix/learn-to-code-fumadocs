# Zustand Introduction

## Learning Objectives

By the end of this lesson, you will:

-  [ ] Bootstrap a Zustand store for TaskFlow Pro preferences.
-  [ ] Understand how Zustand updates state without reducers.
-  [ ] Decide when to prefer Zustand over Context or Redux.
-  [ ] List migration candidates for lightweight shared state.

## Why Zustand?

-  Minimal API: stores are plain functions returning state + actions.
-  Built-in selectors with shallow comparison for performance.
-  Works with server components and asynchronous updates.
-  Coexists with Redux for heavier workflows.

---

## Creating a Store

```tsx
import { create } from "zustand";

type PreferencesState = {
   theme: "light" | "dark";
   density: "comfortable" | "compact";
   setTheme: (theme: PreferencesState["theme"]) => void;
   setDensity: (density: PreferencesState["density"]) => void;
};

export const usePreferencesStore = create<PreferencesState>((set) => ({
   theme: "light",
   density: "comfortable",
   setTheme: (theme) => set({ theme }),
   setDensity: (density) => set({ density }),
}));
```

-  No reducers required—actions update state directly via `set`.
-  Zustand stores support TypeScript inference out of the box.

---

## Using the Store

```tsx
function PreferencesPanel() {
   const { theme, density, setTheme, setDensity } = usePreferencesStore();
   return (
      <div className='space-y-4'>
         <ThemeToggle value={theme} onChange={setTheme} />
         <DensityToggle value={density} onChange={setDensity} />
      </div>
   );
}
```

Zustand components re-render only when selected state changes, reducing
unnecessary updates compared to wide Context providers.
