# Event Handler Best Practices

## Learning Objectives

By the end of this lesson, you will:

-  [ ] Standardise handler naming conventions across the codebase.
-  [ ] Decide when to memoize handlers with `useCallback`.
-  [ ] Capture analytics or debug logs during key interactions.
-  [ ] Document TaskFlow Pro's event handler playbook.

## Project Context

As the interaction surface grows, consistent handler patterns prevent bugs and
make onboarding faster for contributors. This lesson codifies best practices in
a shared document.

---

## Naming Conventions

Use `handle` prefixes for internal functions and `on` prefixes for props.

```tsx
function handleStatusChange(status: TaskStatus) {}
<TaskToolbar onStatusChange={handleStatusChange} />;
```

---

## Memoization Considerations

Memoize handlers when child components rely on reference equality. Otherwise,
prefer simple inline callbacks for readability.

```tsx
const handleSearch = useCallback((value: string) => {
   setFilters((previous) => ({ ...previous, search: value }));
}, []);
```

---

## ‚úÖ Best Practices

### 1. Centralise Analytics Hooks

**Why:** Avoids scattering tracking logic throughout the UI.

```tsx
const track = (event: string, payload: Record<string, unknown>) => {
   if (import.meta.env.DEV) console.info("[analytics]", event, payload);
   // future: send to analytics provider
};
```

### 2. Guard Handlers Against Double Invocation

**Why:** Prevents race conditions when users click rapidly.

```tsx
if (isSubmitting) return;
```

---

## ‚ùå Common Mistakes

### 1. Logging Sensitive Data

**Problem:** Event logs may leak user information.

**Solution:** Scrub or omit sensitive fields before logging.

### 2. Refetching Data on Every Handler

**Problem:** Causes performance problems and flicker.

**Solution:** Mutate local state first; fetch only when necessary.
