# Running the Development Server

## Learning Objectives

By the end of this lesson, you will:

-  [ ] Know how to start and stop the development server
-  [ ] Understand how hot module replacement (HMR) works
-  [ ] Use browser DevTools for debugging
-  [ ] Troubleshoot common development issues

## Project Context

The development server is your primary tool while building TaskFlow Pro. Vite's
lightning-fast HMR means changes appear in your browser instantly, and its
built-in debugging capabilities make it easy to catch errors early.
Understanding how to work efficiently with the dev server will make you a more
productive developer.

## Starting the Development Server

### Basic Commands

```bash
# Start development server
npm run dev

# Alternative (same thing)
npm start

# Start with custom port
npm run dev -- --port 3000

# Start and open browser automatically
npm run dev -- --open

# Start with specific host
npm run dev -- --host 0.0.0.0  # Accessible from network
```

### What Happens When You Start

```
$ npm run dev

  VITE v4.4.9  ready in 423 ms

  ➜  Local:   http://localhost:5173/
  ➜  Network: http://192.168.1.100:5173/
  ➜  press h to show help
```

**Breakdown:**

1. **Vite starts** - Version and startup time shown
2. **Local URL** - Access from your computer
3. **Network URL** - Access from other devices on same network
4. **Keyboard shortcuts** - Press 'h' to see available commands

## Understanding Vite's Development Server

### How It Works

```
Your Code          Vite Server         Browser
─────────         ─────────────        ───────
src/App.jsx  →    Transform JSX   →    Modern JS
src/main.css →    Process CSS     →    CSS Module

                  [No bundling!]
                  [Native ESM]
                  [On-demand compilation]
```

**Key Differences from Traditional Tools:**

-  **No bundling** - Files served individually via ES modules
-  **On-demand** - Only compiles files you request
-  **Fast startup** - No need to bundle entire app
-  **Instant updates** - Hot module replacement (HMR)

### Performance Comparison

```
Traditional Bundler (Webpack):
Start: 15-30 seconds
Rebuild: 2-5 seconds

Vite:
Start: ~400ms
Rebuild: &lt;100ms  ⚡
```

## Hot Module Replacement (HMR)

### What is HMR?

**Hot Module Replacement** updates modules in the browser without a full page
reload, preserving application state.

```jsx
// You're filling out a form
function TaskForm() {
  const [title, setTitle] = useState('Meeting with team');
  const [description, setDescription] = useState('Discuss Q1 goals...');

  return (
    <form>
      <input value={title} onChange={e => setTitle(e.target.value)} />
      <textarea value={description} onChange={e => setDescription(e.target.value)} />
    </form>
  );
}

// You change the button color
<button className="bg-blue-500">Submit</button>
// to
<button className="bg-green-500">Submit</button>

// ✅ With HMR: Button color updates instantly, form data preserved!
// ❌ Without HMR: Full reload, form data lost!
```

### HMR in Action

**Scenario 1: CSS Changes**

```css
/* Change background color */
.app {
   background: #f0f0f0; /* Change to #ffffff */
}
```

Result: Background updates instantly, no reload

**Scenario 2: React Component Changes**

```jsx
// Add a new prop
<TaskCard
   title='Task 1'
   // Add this line:
   priority='high'
/>
```

Result: Component re-renders, state preserved

**Scenario 3: State-Breaking Changes**

```jsx
// Change useState default
const [count, setCount] = useState(0); // to useState(10)
```

Result: Full reload required (state initialization changed)

### HMR Boundaries

```
src/
├── App.jsx              ← Changes here reload whole app
├── main.jsx             ← Changes here reload whole app
├── components/
│   ├── TaskCard.jsx     ← ✅ HMR works great
│   └── TaskList.jsx     ← ✅ HMR works great
└── utils/
    └── helpers.js       ← ⚠️ May require full reload
```

**HMR works best with:**

-  React components
-  CSS/Tailwind classes
-  Component props and JSX

**HMR may not work with:**

-  Top-level state changes
-  Module exports changes
-  Build configuration changes

## ✅ Best Practices

### 1. Keep Terminal Visible

**Why:** Catch errors immediately

```bash
# ✅ Good: Terminal visible while coding
┌──────────────┬──────────┐
│              │          │
│  Code        │ Browser  │
│              │          │
├──────────────┴──────────┤
│  $ npm run dev          │
│  ✓ Ready in 423ms       │
└─────────────────────────┘

# ❌ Bad: Hidden terminal
# You miss build errors!
```

### 2. Use Browser DevTools Effectively

**Why:** Faster debugging

```javascript
// ✅ Good: Descriptive logs
console.log("TaskCard rendered:", { id, title, status });
console.warn("API response slow:", responseTime);

// ❌ Bad: Unclear logs
console.log(task); // What is this?
console.log("here"); // Where is "here"?
```

### 3. Save Files Frequently

**Why:** HMR triggers on save

```
Cmd+S (Mac) / Ctrl+S (Windows)

Configure VS Code Auto Save:
File → Auto Save (check it!)

Or: Set delay in settings:
"files.autoSave": "afterDelay",
"files.autoSaveDelay": 1000
```

## ❌ Common Mistakes

### 1. Not Restarting After Config Changes

**Problem:** Changes to config files need server restart

```javascript
// Changed vite.config.js
export default defineConfig({
  server: {
    port: 3000,  // Changed from 5173
  },
});

// ❌ Bad: Keep server running
// Still on port 5173!

// ✅ Good: Restart server
// Ctrl+C to stop
npm run dev
// Now on port 3000 ✓
```

**Files that require restart:**

-  `vite.config.js`
-  `package.json`
-  `.env` files
-  `tailwind.config.js`
-  `postcss.config.js`

### 2. Ignoring Console Errors

**Problem:** Errors break functionality silently

```bash
# Browser Console:
Warning: Each child in a list should have a unique "key" prop.

# ❌ Bad: Ignore warning
# App works but may have bugs later

# ✅ Good: Fix immediately
{tasks.map(task => (
  <TaskCard key={task.id} {...task} />
))}
```

### 3. Port Already in Use

**Problem:** Another process using the port

```bash
# Error message:
Port 5173 is in use, trying another one...
Port 5174 is in use, trying another one...

# ❌ Bad: Keep trying random ports

# ✅ Good: Kill the other process
# macOS/Linux:
lsof -ti:5173 | xargs kill -9

# Or: Use different port
npm run dev -- --port 3000
```

## Troubleshooting Guide

### Issue: Server Won't Start

```bash
# Check Node version
node --version  # Should be 18+

# Clear npm cache
rm -rf node_modules
npm install

# Check for port conflicts
lsof -i :5173  # macOS/Linux
netstat -ano | findstr :5173  # Windows
```

### Issue: Changes Not Reflecting

```bash
# 1. Hard refresh browser
Cmd+Shift+R (Mac)
Ctrl+Shift+R (Windows)

# 2. Clear Vite cache
rm -rf node_modules/.vite

# 3. Restart dev server
# Stop (Ctrl+C) and restart
npm run dev
```

### Issue: Page is Blank

```javascript
// Check browser console for errors
// Common causes:

// 1. Syntax error
const Component = () => {
  return (
    <div>  {/* Missing closing tag */}
  );
};

// 2. Import error
import { App } from './App';  // Should be default import

// 3. Render error
ReactDOM.createRoot(document.getElementById('root')).render(<App />);
// Check if element exists!
```

### Issue: Slow Performance

```bash
# Check what's running
top  # macOS/Linux
taskmgr  # Windows

# Reduce memory usage:
# - Close unused tabs
# - Restart VS Code
# - Disable unnecessary extensions

# Optimize Vite:
# vite.config.js
export default defineConfig({
  server: {
    watch: {
      ignored: ['**/node_modules/**', '**/.git/**'],
    },
  },
});
```
